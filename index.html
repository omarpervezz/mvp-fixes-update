⚡ Fix 4 — Performance & Stability

Fixed 4 issues

Issue & Root Cause

Remote places (e.g., “Whitetop, VA 24292, USA”) felt slow on first search due to upstream API latency.

What I Changed

  • LRU cache: I store normalized queries in memory; the cache auto-evicts least-recent entries to keep memory bounded. Warm searches are instant (“CACHE HIT”).
  • In-flight de-dupe: Collapses duplicate concurrent queries into one network call.
  • Responsive UI: Inputs stay interactive; Get 3 Leads gates on valid selection.

Verification & Impact

  • Cold: one POST (~361ms) to Places; results returned.
  • Warm: served from cache; no extra network calls.
  • Rapid re-queries: single upstream call shared by all.
  • Outcome: faster repeats, fewer external calls, better perceived speed.

🔒 Security — Assurance

Verified

What I implemented

  • Server-side Places calls (Next.js Server Actions): API key in env; never sent to client.
  • No sensitive logs: Keys/tokens scrubbed; only non-PII metadata logged.
  • Minimal payloads: Client sends query + session token; server returns normalized suggestions.
  • Defense-in-depth: Env-only keys, CORS restricted, memory-only caching.

How I verified

  • DevTools Network: POST to /fix with small JSON body; responses show no credentials.
  • No API key in headers/params/source; console has no secret data.
  • Result: No client-side key exposure; reduced attack surface.
